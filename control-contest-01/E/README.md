# E. Очередь в Дринкит

## Условие

Дана перестановка `p[1..n]`.

Разрешён только такой обмен:

- выбрать позиции `i` и `j`,
- на позиции `i` обязательно должен стоять элемент `1`, то есть `p[i] = 1`,
- расстояние должно удовлетворять:

```text
|i - j| <= floor(n / 2)
````

после чего элементы на позициях `i` и `j` меняются местами.

Нужно либо:

* получить перестановку

```text
1, 2, ..., n
```

не более чем за `4n` обменов,

либо вывести `-1`, если это невозможно.


# Главная идея задачи

Единственный элемент, который умеет “двигать” другие элементы, — это `1`.

То есть любой допустимый обмен всегда выглядит так:

> позиция, где сейчас стоит `1`, меняется с некоторой другой позицией на расстоянии не больше `n / 2`.

Поэтому задача сводится к тому, чтобы научиться с помощью `1`:

1. перемещаться по массиву;
2. ставить остальные элементы на нужные места;
3. делать это в ограничении по числу операций.


# Что на самом деле происходит

Пусть сейчас `1` стоит в позиции `pos1`.

Тогда за один ход мы можем переместить `1` только в позиции из диапазона:

```text
[pos1 - floor(n/2), pos1 + floor(n/2)]
```

То есть `1` не может “прыгнуть” куда угодно за один обмен, но может двигаться по массиву достаточно далеко.

Именно это используется в решении:

* сначала `1` доводят до правого края,
* потом с его помощью “вытаскивают” нужные элементы на свои места,
* постепенно сортируя массив.


# Когда задача невозможна

## Случай `n = 3`

Тогда:

```text
floor(n / 2) = 1
```

То есть `1` может меняться только с соседней позицией.

Не все перестановки длины 3 можно так отсортировать.

В коде отдельно проверяются три плохих случая:

```text
[1, 3, 2]
[3, 1, 2]
[3, 2, 1]
```

Если перестановка ровно такая, ответ `-1`.
Это видно в коде через функцию `badTriple`.

Идея тут в том, что при `n = 3` граф достижимых состояний слишком маленький, и именно эти конфигурации не приводятся к `[1,2,3]`.


# Общая стратегия

Решение строится конструктивно.

Оно не просто проверяет возможность, а **явно строит последовательность обменов**.


## Шаг 1. Довести `1` до конца массива

Пусть `ind` — текущая позиция элемента `1`.

Пока `1` не дошёл до последней позиции:

* если можно прыгнуть на `half = floor(n/2)` вправо — прыгаем;
* иначе прыгаем сразу в конец.

Так `1` оказывается в правом конце массива.

### Зачем это нужно

Когда `1` стоит справа, им удобно начинать “разбирать” массив с конца и постепенно ставить элементы по местам.


## Шаг 2. Двигаться влево и сортировать элементы

Дальше алгоритм работает итеративно.

Он использует текущую позицию `1` и пытается:

* доставить нужный элемент в правильную позицию,
* затем сдвинуть границу уже отсортированной части.

По сути идёт по массиву и шаг за шагом делает так, чтобы всё больше префикс/суффикс оказывался на своих местах.


# Почему нужен массив позиций `p`

В коде хранится:

* `a[pos]` — кто стоит в позиции `pos`,
* `p[val]` — в какой позиции сейчас стоит значение `val`.

Это очень важно, потому что после каждого обмена нужно быстро знать:

* где сейчас находится `1`,
* где стоит нужный элемент,
* куда именно нужно его двигать.

Без массива `p` поиск каждого значения занимал бы `O(n)`, и решение стало бы слишком медленным.

С `p` каждая такая операция делается за `O(1)`.


# Что делает первый цикл

```text
while ind < n - 1
```

Здесь `ind = p[1]` — позиция единицы.

Алгоритм двигает `1` вправо:

* либо на `half`,
* либо сразу в `n - 1`.

Это законный обмен, потому что:

* меняется позиция, где стоит `1`,
* расстояние не больше `half`.

После этого `1` гарантированно оказывается в конце.


# Что делает основной большой цикл

```text
while ind > 0
```

Пока `1` не оказался на первой позиции, продолжаем работу.

Внутри есть две важные фазы.


## Фаза A. Спустить `1` к нужной позиции

Считается:

```text
to = p[ind + 1]
```

Это означает: мы смотрим, где стоит элемент, который должен быть “рядом” с текущей границей.

Дальше `1` двигается влево:

* либо прыжком на `half`,
* либо прямо в нужную позицию `to`.

Эта часть нужна, чтобы подвести `1` к месту, где можно начать локально переставлять элементы правильно.


## Фаза B. Продвинуть элемент в нужную зону

После того как `1` оказался в подходящем месте, начинается вторая часть:

```text
while ind <= n - j - 1
```

Здесь алгоритм либо:

* продвигает `1` вправо контролируемыми прыжками,
* либо делает пару локальных обменов, если нужно аккуратно обойти ограничение на расстояние.

Именно тут происходит основная “сборка” правильного порядка.


# Интуитивное объяснение

Полезно думать о `1` как о специальном “манипуляторе”.

Он не сортирует массив сам по себе, но позволяет:

* приехать в некоторую позицию,
* взять элемент оттуда,
* перевезти его в другую допустимую область,
* повторять это, пока элементы не окажутся где надо.

Так как разрешённый радиус каждого обмена равен `floor(n/2)`, `1` может достаточно быстро покрывать массив.


# Почему ограничение `4n` выполняется

Алгоритм специально построен так, чтобы число операций было линейным.

Интуитивно:

* `1` проходит по массиву крупными прыжками;
* на каждом этапе либо продвигается граница сортировки,
* либо выполняется ограниченное число локальных исправлений.

В конце код ещё дополнительно **сокращает ответ**:

если подряд встретились два взаимно обратных обмена:

```text
(x, y), (y, x)
```

то они просто удаляются, потому что ничего не меняют.

Это уменьшает итоговое количество шагов.

После этого проверяется:

```text
если число операций > 4n → вывести -1
иначе вывести ответ
```


# Зачем удалять взаимно обратные операции

Если мы сделали:

```text
swap(i, j)
swap(j, i)
```

то итоговый эффект нулевой.

Такие пары можно безопасно выбросить.

В коде это делается проходом по массиву ответов.

Это не меняет корректность, но уменьшает длину ответа.


# Почему все операции корректны

Для каждого добавляемого обмена всегда соблюдается условие задачи:

## 1. На одной из выбранных позиций стоит `1`

Алгоритм всегда отслеживает текущую позицию `1` через `p[1]` и строит обмены именно от неё.

## 2. Расстояние не больше `floor(n/2)`

Все переходы устроены так, что используются:

* либо сдвиги на `half`,
* либо меньшие локальные шаги,
* либо переходы к специально выбранным близким позициям.

То есть запрещённых длинных обменов алгоритм не делает.


# Почему решение работает быстро

Во всех операциях используются только:

* обращения по индексу,
* обновления массивов,
* добавление ответа.

Нет сортировок, нет деревьев, нет поиска по массиву.

Поэтому на один тест:

```text
O(n)
```

Так как сумма всех `n` по тестам не превосходит `2 * 10^5`, решение проходит по времени.


# Разбор на примере

## Пример

```text
n = 4
p = [3, 4, 2, 1]
```

`half = 2`.

Изначально `1` стоит в позиции `4`.

Нужно получить:

```text
[1, 2, 3, 4]
```

Один из допустимых ответов:

```text
4 2
2 3
3 1
```

Проверим:

### После `4 2`

Меняем позиции 4 и 2:

```text
[3, 1, 2, 4]
```

### После `2 3`

Меняем позиции 2 и 3:

```text
[3, 2, 1, 4]
```

### После `3 1`

Меняем позиции 3 и 1:

```text
[1, 2, 3, 4]
```

Готово.

Во всех шагах:

* в первой позиции обмена стоит `1`,
* расстояние не больше `2`.


# Почему идея с “довести 1 до края” полезна

Когда `1` находится у края, у него максимальная зона достижимости в одну сторону и удобнее контролировать перестановки.

Это типичный приём в конструктивных задачах с ограниченными обменами:

* сначала перемещаем специальный элемент в “удобную” позицию,
* потом уже используем его как инструмент.


# Структуры данных

## Массив `a`

Хранит текущую перестановку:

```text
a[pos] = значение в позиции pos
```

## Массив `p`

Хранит обратное соответствие:

```text
p[val] = позиция значения val
```

То есть:

* `a` отвечает на вопрос “кто стоит здесь?”,
* `p` отвечает на вопрос “где стоит этот элемент?”.

Это стандартная и очень удобная пара структур для задач про перестановки и обмены.



# Корректность — на уровне идеи

Полное формальное доказательство у этого решения довольно громоздкое, но логика такая:

## Лемма 1

Если алгоритм добавляет обмен, то он допустим.

**Почему:**
обмен всегда строится через текущую позицию элемента `1`, а длина перехода контролируется и не превосходит `floor(n/2)`.


## Лемма 2

Каждый крупный этап либо:

* перемещает `1` ближе к нужной зоне,
* либо увеличивает уже корректно организованную часть массива.

**Почему:**
индексы и границы в циклах монотонно двигаются, алгоритм не зацикливается.


## Лемма 3

Алгоритм завершает работу за линейное число шагов.

**Почему:**
границы двигаются только в одну сторону, а на каждом этапе число вспомогательных обменов ограничено константой.


## Следствие

Если алгоритм не выводит `-1`, то построенная последовательность действительно переводит перестановку в

```text
[1, 2, ..., n]
```


# Частые ошибки при попытке решить самому

## 1. Пытаться менять любые два элемента

По условию можно менять только позицию, где стоит `1`, с другой позицией.

Это очень жёсткое ограничение.


## 2. Забыть, что расстояние ограничено

Недостаточно просто “хотеть” поставить элемент на место.
Нужно, чтобы каждый шаг был длины не больше `floor(n/2)`.


## 3. Искать кратчайший путь

Это не требуется.
Нужно просто построить **какой-то** путь длиной не больше `4n`.


## 4. Не хранить обратный массив позиций

Без `p[val]` обновления становятся слишком медленными.


## 5. Не обрабатывать маленький случай `n = 3`

Именно там появляются недостижимые состояния.


# Асимптотика

Для одного теста:

```text
O(n)
```

По памяти:

```text
O(n)
```

По всем тестам:

```text
O(sum n)
```

что укладывается в ограничение:

```text
sum n <= 2 * 10^5
```


# Что делает постобработка ответа

После построения списка операций алгоритм проходит по нему и удаляет соседние взаимно обратные пары:

```text
(i, j), (j, i)
```

Это:

* не меняет результат,
* может уменьшить число операций,
* помогает уложиться в `4n`.


# Краткий вывод

Это конструктивная задача про сортировку перестановки при очень необычной операции:

* обмен возможен только через элемент `1`,
* длина обмена ограничена `floor(n/2)`.

Ключевая идея решения:

* использовать `1` как подвижный “инструмент”,
* сначала довести его до удобной позиции,
* потом шаг за шагом расставлять элементы по местам,
* в конце убрать лишние обратные операции.

Именно так получается либо построить корректную последовательность обменов длиной не больше `4n`, либо определить, что для данного случая это невозможно.

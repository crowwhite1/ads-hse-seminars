# B. Дедлайны и/или паника

## Идея задачи

Нам дана интерактивная структура данных, про которую известно только одно: это либо

- **стек** (`stack`),
- либо **очередь** (`queue`).

Нужно определить, какой именно тип структуры перед нами.

Доступны запросы двух видов:

- `put x` — добавить элемент `x`,
- `get` — извлечь следующий элемент.

Если структура пуста, `get` возвращает `-1`.

Также известно важное ограничение:

- в структуре в любой момент находится **не более 297 элементов**.


## Ключевое наблюдение

Мы не знаем, что уже лежит внутри структуры в начале работы программы.

Поэтому сначала нужно привести структуру к **известному состоянию**, то есть гарантированно **очистить** её.

Так как элементов не больше `297`, достаточно сделать:

- `297` запросов `get`.

После этого структура точно пуста, потому что:

- если в ней было меньше 297 элементов, лишние `get` будут просто возвращать `-1`,
- если было ровно 297 элементов, мы извлечём их все.


## Как отличить стек от очереди

Когда структура уже пуста, делаем:

1. `put 3`
2. `put 4`
3. `get`

Теперь возможны два варианта:

### Если это очередь

Очередь работает по принципу **FIFO** (*first in, first out* — первый вошёл, первый вышел).

Мы добавили сначала `3`, потом `4`, значит первым выйдет `3`.

### Если это стек

Стек работает по принципу **LIFO** (*last in, first out* — последний вошёл, первый вышел).

Мы добавили сначала `3`, потом `4`, значит первым выйдет `4`.


## Алгоритм

1. 297 раз отправить `get` и считать ответ.
2. Отправить `put 3`.
3. Отправить `put 4`.
4. Отправить `get`.
5. Если получили:
   - `3` → вывести `queue`
   - `4` → вывести `stack`


## Почему это корректно

Докажем корректность алгоритма.

### Шаг 1. После 297 запросов `get` структура пуста

По условию в структуре не более 297 элементов.

Каждый запрос `get` либо:
- удаляет один существующий элемент,
- либо возвращает `-1`, если структура уже пуста.

Значит после 297 запросов все возможные элементы точно будут удалены.


### Шаг 2. После `put 3`, `put 4` структура содержит ровно два элемента

Так как до этого структура была очищена, после двух операций добавления внутри находятся только:

- сначала `3`,
- затем `4`.


### Шаг 3. Один запрос `get` однозначно определяет тип структуры

- Для **очереди** первым извлекается самый старый элемент, то есть `3`.
- Для **стека** первым извлекается самый новый элемент, то есть `4`.

Следовательно, по ответу на этот запрос мы всегда однозначно определяем тип структуры.


## Оценка числа запросов

Используется:

- `297` запросов на очистку,
- `2` запроса `put`,
- `1` запрос `get` для проверки.

Итого:

**300 запросов**.

Это ровно укладывается в ограничение задачи.


## Сложность

### По числу запросов
- **300**

### По памяти
- **O(1)**


## Пример работы идеи

Допустим, после очистки структура пуста.

Отправляем:

```text
put 3
put 4
get
````

### Если это очередь:

```text
3
```

### Если это стек:

```text
4
```

## Особенности интерактивной задачи

После каждого запроса нужно:

* вывести перевод строки,
* **сбросить буфер вывода**.

Иначе интерактор не увидит запрос, и решение может получить вердикт `IL`.

## Вывод

Решение строится на двух простых идеях:

1. **сначала полностью очистить структуру**, используя ограничение в 297 элементов;
2. **положить два разных элемента** и посмотреть, какой из них вернётся первым.

Это даёт простое, строго корректное решение, которое использует ровно допустимое число запросов.
